🗺️ The AI Landscape — A Quick Map

Artificial intelligence is everywhere — but the terminology can be confusing. Is machine learning the same as AI? What's the difference between a model and a product? This guide gives you a clear mental model to understand how it all fits together.

💡
TPM analogy: Think of AI like a Russian nesting doll. AI is the outer shell — the big umbrella. Inside it is Machine Learning. Inside ML is Deep Learning. And inside Deep Learning are Large Language Models (like ChatGPT). Each layer is more specific than the one surrounding it.

Two things people often confuse: Products vs. Models

📱

AI Products (Apps)

The interface you interact with. Examples: ChatGPT (chat app), Codex (coding agent), Claude.ai (chat app), Claude Code (coding agent).

⚙️

AI Models (Engines)

The underlying AI engine doing the work. Examples: GPT-4o, Codex-mini, Claude Sonnet, Claude Opus. Accessible via APIs.

🔑
Key insight: ChatGPT is to GPT-4 as a car is to its engine. You drive the car (use the product), but the engine (model) is what actually makes it go. Developers can access the engine directly via APIs to build their own products.

Key Takeaways

  • ✅ AI → ML → Deep Learning → LLMs: each is a subset of the previous
  • ✅ Products (ChatGPT, Claude Code) are applications built on top of models
  • ✅ Models (GPT-4, Claude Sonnet) are the AI engines accessible via APIs
  • ✅ The same model can power multiple products (e.g., Claude Sonnet powers Claude Code)

📊 What is Machine Learning?

Machine Learning (ML) is a way of building software where, instead of writing rules by hand, you give the system examples and let it figure out the patterns.

💡
TPM analogy: Imagine you want to build a spam filter. The traditional approach: write rules like "if email contains 'free money', it's spam." The ML approach: show the system 100,000 labeled emails (spam / not spam) and let it learn the patterns itself. The ML model will discover patterns you never thought to write.

Three types of Machine Learning

🏷️

Supervised Learning

Train on labeled examples. The model learns to predict outputs from inputs. (Email spam detection, image classification, price prediction)

🔍

Unsupervised Learning

Find hidden patterns in unlabeled data. The model groups similar things together. (Customer segmentation, anomaly detection)

🎮

Reinforcement Learning

Learn by trial and error with rewards and penalties. (Game-playing AI, robotics, recommendation systems)

ML in the real world — examples you know

🎬 Netflix recommendations
Supervised + collaborative filtering ML predicts what you'll want to watch next based on your history and similar users.
📧 Spam filters
Trained on millions of spam/not-spam examples. Updates continuously as spammers evolve their tactics.
🏦 Fraud detection
ML models flag unusual transactions in real time by learning patterns from historical fraud cases.
🔍 Search ranking
Google's ranking is heavily ML-driven — learning from billions of queries what results people actually find useful.

Key Takeaways

  • ✅ ML learns from data examples instead of hand-coded rules
  • ✅ Three types: Supervised (labeled), Unsupervised (unlabeled), Reinforcement (trial & error)
  • ✅ Data quality is the #1 factor in ML success
  • ✅ ML models need monitoring after deployment — data can drift over time

🧠 What is Deep Learning?

Deep Learning is a type of machine learning that uses artificial neural networks — systems loosely inspired by the human brain. The "deep" refers to the many layers in these networks.

💡
TPM analogy: Imagine a photo recognition system. A traditional ML model might need you to manually extract features like "has four legs" or "has fur." A deep learning model learns to extract its own features automatically — starting with edges and shapes in early layers, then assembling those into ears and eyes, then into "cat." No manual feature engineering required.

Why Deep Learning is a breakthrough

🔧

Traditional ML

You must manually define which features to extract from data. Requires significant domain expertise.

🧠

Deep Learning

Learns to extract relevant features automatically. Scales dramatically better with more data.

Key Deep Learning architectures (you'll hear these terms)

🔄 CNN (Convolutional Neural Network)
Specialized for images. Learns spatial patterns. Powers face recognition, medical imaging, self-driving vision.
📝 RNN / LSTM
Designed for sequences (text, time series). Early approach to language tasks — mostly replaced by Transformers.
🔀 Transformer
The architecture behind all modern LLMs. Uses "attention" to understand context across long sequences. Introduced in 2017 ("Attention Is All You Need").
🎨 Diffusion Model
Powers image generation (DALL-E, Stable Diffusion, Midjourney). Learns to progressively "denoise" random noise into images.

Key Takeaways

  • ✅ Deep learning = ML with many-layer neural networks
  • ✅ "Deep" = many layers, not depth of understanding
  • ✅ Learns features automatically — no manual feature engineering
  • ✅ The Transformer architecture (2017) is what powers all modern LLMs
  • ✅ Needs lots of data and compute — but scales extremely well

💬 Large Language Models (LLMs)

A Large Language Model (LLM) is a deep learning model trained on enormous amounts of text. It learns to understand and generate language by predicting the next word (or "token") in a sequence — over and over, at massive scale.

💡
TPM analogy: Imagine reading every book, article, website, and code repository ever written — billions of pages. After reading all of that, you develop an incredibly rich intuition for language patterns, facts, reasoning, and writing styles. That's what an LLM has done, except it "read" everything in a compressed mathematical form.

What are "tokens"?

LLMs don't process whole words — they work with tokens, which are roughly word-pieces. Here's how the sentence "The bank by the river was flooded" gets split:

What LLMs can (and cannot) do

✅

LLMs are great at

Writing & editing · Summarization · Q&A · Code generation · Translation · Brainstorming · Following instructions · Conversation

⚠️

LLMs struggle with

Precise math · Reliable factual recall (can "hallucinate") · Real-time information · Counting · Multi-step logical chains

Key LLM concepts you'll hear

📐 Parameters / Weights
The learned numbers inside the model. GPT-4 has hundreds of billions of parameters. More parameters ≠ always better, but generally more capable.
🎯 Prompt Engineering
The art of crafting inputs to get the best outputs from an LLM. Clear, specific prompts with context produce dramatically better results.
🔧 Fine-tuning
Taking a pre-trained base model and training it further on a specific dataset to specialize it (e.g., medical Q&A, customer service).
📚 RAG (Retrieval-Augmented Generation)
Combining an LLM with a search system that retrieves relevant documents. Solves the "outdated knowledge" problem by injecting fresh context.
🧲 Context Window
How much text an LLM can "see" at once (input + output). GPT-4 supports 128K tokens; Claude 3 supports up to 200K tokens (~150,000 words).
🤖 AI Agents
LLMs that can take actions: browse the web, run code, call APIs, manage files. Coding agents like Codex and Claude Code are examples.

Key Takeaways

  • ✅ LLMs are deep learning models trained on massive text datasets
  • ✅ They work by predicting the next token — this scales to powerful capabilities
  • ✅ Attention mechanism lets them understand context across long sequences
  • ✅ Strong at language tasks; can "hallucinate" — always verify factual claims
  • ✅ Context window = how much text the model can process at once

Questions every TPM should ask at project kickoff

🗄️ About the data
"Does the labeled training data exist today?" · "Who owns data access and what are the privacy/legal constraints?" · "How will data drift be detected in production?"
🎯 About success
"What does 'good enough' look like, and who decides?" · "What's the baseline we're beating?" · "Is this a launch blocker or a post-launch improvement target?"
⚠️ About risk
"What's our rollback plan if the model degrades?" · "Which user groups could be harmed by a wrong prediction?" · "Is there a human-in-the-loop safety net?"
👥 About the team
"Who is the final decision-maker on model quality?" · "How will research hand off to engineering?" · "Who monitors production performance post-launch?"

Key Takeaways

  • ✅ Data delays are the #1 cause of ML project slippage — track data readiness as a milestone
  • ✅ Challenge accuracy targets upfront — they're often aspirational, not researched
  • ✅ Build "productionization time" explicitly into every AI project plan
  • ✅ "Buy before build" — integrating an API is often faster and more reliable than training a custom model
  • ✅ Define rollback criteria before deploying — not after something goes wrong

🔄 AI Project Lifecycle & MLOps

A production ML system is much more than a trained model — it's a pipeline of data, code, experiments, deployment infrastructure, and monitoring. MLOps (Machine Learning Operations) is the discipline of making that pipeline reliable, repeatable, and maintainable.

💡
TPM analogy: Shipping a trained model to production is like launching a product — but the product never stops changing. The model needs to be retrained as data evolves, re-evaluated as requirements shift, and monitored like a live service. Think of MLOps as DevOps for models.

Key MLOps concepts you'll hear

🔀 CI/CD for ML
Continuous Integration/Deployment applied to ML: automatically retrain, evaluate, and deploy updated models when new data arrives or code changes.
📈 Data Drift
When real-world input data shifts away from the training distribution. Example: a fraud model trained pre-COVID may miss new fraud patterns that emerged during the pandemic.
🏷️ Model Registry
A versioned catalog of trained models — similar to a code repository, but for model artifacts, metadata, and evaluation results. Tools: MLflow, Vertex AI Model Registry.
🪄 Feature Store
A centralized repository for precomputed ML features, ensuring consistency between training and serving. Prevents "training-serving skew" — a common production bug.
🐦 Shadow Mode / Canary
Shadow mode: run a new model in parallel without affecting users (log predictions only). Canary: route a small % of real traffic to the new model before full rollout.
🔁 Retraining Triggers
Conditions that kick off a new training run: scheduled (weekly), data-volume threshold, or metric-based (accuracy drops below X%). Agree on triggers before launch.

Agile for AI: Where traditional sprints need adaptation

📅

Traditional Agile

• Predictable 2-week sprints
• Clear acceptance criteria
• "Done" is a working feature
• Velocity is measurable

🧪

AI/ML Adaptation Needed

• Experiments may need 2–6 weeks
• Success criteria are probabilistic
• "Done" includes production monitoring
• Value is delivered non-linearly

Key Takeaways

  • ✅ ML projects are circular, not linear — expect to cycle back through phases
  • ✅ Data work takes 60–80% of the timeline; plan accordingly
  • ✅ MLOps = DevOps for ML — deployment, monitoring, and retraining are ongoing
  • ✅ Shadow mode and canary releases reduce risk of model failures at launch
  • ✅ Define retraining triggers before launch — not after the model degrades

🗣️ Stakeholder Communication & Translation

A core TPM superpower is translating between technical and non-technical stakeholders. In AI projects, this means explaining complex concepts in plain English, managing expectations about what AI can and can't do, and framing technical tradeoffs as business decisions.

💡
TPM analogy: You're the interpreter between two groups speaking different languages. Engineers talk about precision, recall, and latency p99. Executives talk about revenue impact, user trust, and risk. Your job is to translate — accurately — in both directions.

Jargon Translator: Technical → Plain English

🎯 Model accuracy
They say: "We achieved 89% accuracy." You say to execs: "The model gets the right answer about 9 times out of 10 on our test set." (But also: ask what the baseline is — 89% could be bad if the most common class is 90% of data.)
⚖️ Precision vs. Recall
Precision: "When we flag something, how often are we right?" Recall: "Of all the real cases, how many did we catch?" There's a tradeoff — tightening one loosens the other. Which matters more is a business decision.
💤 Model Latency
They say: "P99 latency is 450ms." You say: "For 99% of requests, the AI responds in under half a second. The slowest 1% take longer — this affects our SLA commitments."
🌀 Overfitting
They say: "The model is overfit." You say: "The model memorized the training examples instead of learning generalizable patterns. It scores great on tests but fails on real-world data."
🎲 Confidence Score
They say: "We'll surface predictions above 0.85 confidence." You say: "We'll only show AI recommendations when the model is 85%+ sure — lower-confidence predictions go to manual review."
🔄 Retraining
They say: "We need to retrain the model quarterly." You say: "We update the AI every 3 months with new data so it stays current and doesn't degrade as the world changes."
🌫️ Hallucination
They say: "The LLM hallucinated." You say: "The AI confidently gave us wrong information it made up. For this use case, we need human review of all AI outputs before they're used."
📚 RAG
They say: "We're using RAG for the knowledge base." You say: "We're connecting the AI to our internal documents so it answers based on our actual data, not just its general training."

Managing AI Expectations: What to say when…

🚫

Don't say this

  • ❌ "The AI will handle everything automatically"
  • ❌ "It's like ChatGPT but for us"
  • ❌ "The model is 95% accurate so we're good"
  • ❌ "We just need to train it on our data"
  • ❌ "AI will be done in 3 weeks"
✅

Say this instead

  • ✅ "It will automate X, with human review for edge cases"
  • ✅ "It uses similar technology, scoped to this specific task"
  • ✅ "95% on the test set — let's see how it holds up on live data"
  • ✅ "Data collection, cleaning, and labeling adds 6–12 weeks"
  • ✅ "Experimentation phase will take 4–8 weeks — results will inform timeline"

Metrics that matter: Technical vs. Business KPIs

🔬

Technical Metrics

  • • Accuracy, Precision, Recall, F1 score
  • • AUC-ROC (ranking quality)
  • • Latency (p50, p95, p99)
  • • Training time, inference cost per request
  • • Data drift metrics
📈

Business KPIs

  • • Time saved (hours/week automated)
  • • Error rate reduction (vs. manual process)
  • • Revenue impact or cost savings ($)
  • • User satisfaction / NPS change
  • • Decision quality improvement

Key Takeaways

  • ✅ Always translate technical metrics into business impact for executive audiences
  • ✅ Precision vs. recall tradeoff is a business decision — present it as one
  • ✅ Set expectations early: AI projects require data work that takes weeks, not days
  • ✅ "95% accuracy" is meaningless without a baseline to compare against
  • ✅ Hallucination risk means high-stakes AI outputs need human review workflows

✍️ Prompt Engineering

Prompt engineering is the practice of crafting inputs to LLMs to get the most accurate, useful, and reliably formatted outputs. It's part art, part science — and an increasingly valuable skill for TPMs who use AI tools or manage teams that build them.

💡
TPM analogy: Writing a good prompt is like writing a good requirements document. Vague requirements produce vague output. Specific requirements — with context, constraints, and examples — produce reliable, usable output. The more precise you are about what you want, the better the AI can deliver it.
🔑
Why TPMs need this: Whether you're using AI tools personally, reviewing AI-powered features, or working with teams integrating LLM APIs, understanding how prompts work helps you set realistic expectations, debug outputs, and make better product decisions.

Prompt engineering best practices

📌

Be specific

Include context, constraints, audience, and desired format. Vague prompts produce vague output.

🔁

Iterate

Treat prompting like debugging. Change one variable at a time and observe the effect.

🧪

Evaluate systematically

Don't judge a prompt on one output. Test it on diverse inputs to find failure modes.

TPM-specific use cases for prompt engineering

📝 Status reports
Paste meeting notes into an LLM with a structured output prompt to auto-generate weekly status updates in your team's format.
⚠️ Risk identification
Share a project plan and ask the model to "identify risks, dependencies, and missing requirements as a senior PM would." Use as a pre-mortem tool.
🗣️ Stakeholder prep
Role-prompt the model as a skeptical executive and have it challenge your roadmap — great prep for tough review meetings.
📊 Data summarization
Summarize long technical documents, PRDs, or research papers into 3-bullet executive summaries. Chain-of-thought helps for complex synthesis.

Key Takeaways

  • ✅ Prompt quality directly determines output quality — invest in crafting good prompts
  • ✅ Few-shot examples are the fastest way to improve output format and consistency
  • ✅ "Think step by step" (chain-of-thought) significantly improves reasoning tasks
  • ✅ Requesting structured output (JSON, tables) makes LLM output easier to use programmatically
  • ✅ Treat prompt engineering iteratively — one change at a time, test on diverse inputs

⚖️ Risk, Ethics & Responsible AI

AI systems can cause real harm — through biased decisions, privacy violations, safety failures, or misuse. As a TPM, you're often the person responsible for asking "should we?" when everyone else is focused on "can we?". Responsible AI isn't just ethics — it's also risk management.

💡
TPM analogy: Imagine shipping a product without security testing because it "works in the demo." Responsible AI is the security testing equivalent for AI systems — it's not optional for production, it's a standard part of the launch checklist.

Types of AI risk

⚖️

Bias & Fairness

A model trained on historical data inherits historical biases. Can produce discriminatory outcomes in hiring, lending, criminal justice, and healthcare.

🔒

Privacy & Security

Models can memorize training data and inadvertently leak PII. Adversarial attacks (prompt injection, data poisoning) can subvert model behavior.

🌫️

Reliability & Safety

Hallucinations, distributional shift, and edge case failures. In high-stakes domains (medical, legal, safety-critical), wrong outputs can cause serious harm.

Bias: What TPMs need to know

📊 Training data bias
If historical data reflects past discrimination (e.g., hiring data with gender skew), the model will learn and replicate those patterns. Garbage in = biased model out.
🔍 Representation bias
When certain groups are underrepresented in training data, the model performs worse on them. A face recognition system trained mostly on light-skinned faces performs poorly on dark-skinned faces.
📏 Measurement bias
Using a proxy metric that's systematically unfair. Using zip code as a credit risk proxy can perpetuate redlining — zip code correlates with race in the US.
🧪 Evaluation bias
Only testing performance on aggregate metrics hides disparate impact. Always disaggregate evaluation by demographic slices relevant to your use case.

Regulatory landscape

🇪🇺 EU AI Act (2024)
World's first comprehensive AI law. Risk-based tiers: Unacceptable risk (banned), High risk (strict requirements — hiring, credit, healthcare), Limited/Minimal risk. High-risk AI requires conformity assessments, transparency, and human oversight.
🇺🇸 US Executive Order on AI (2023)
Requires safety testing for powerful AI models, standards for watermarking AI content, and guidance for agencies using AI in government services. Sector-specific rules from FTC, EEOC, and others.
🔐 GDPR & Data Privacy
EU regulation requiring lawful basis for data use, right to explanation for automated decisions, and data minimization. Applies to ML training data and AI-driven decisions about EU residents.
🏥 Sector regulations
HIPAA (healthcare), FCRA (credit), EEOC (employment) apply existing rules to AI. If your AI touches one of these domains, involve legal and compliance early — not at launch.

  • ✅ High-stakes AI decisions (hiring, lending, healthcare) face specific legal requirements
  • ✅ Explainability is not optional — affected users and regulators may have a right to explanations
  • ✅ The EU AI Act (2024) classifies AI by risk — know which tier your product falls in
  • ✅ Responsible AI is a launch requirement, not an afterthought — build it into the project plan
